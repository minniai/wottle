## Implementation Order

Implement in tier order. Each tier is independently push-able and testable in CI before the next tier begins.

### Tier 1 (FR-001–FR-005) — Start here

These are all safe, low-risk changes that improve cache hits and remove serial constraints.

1. **FR-004**: Change all `pnpm install --frozen-lockfile false` → `pnpm install --frozen-lockfile` in every job (7 occurrences)
2. **FR-001**: Add `cache: "pnpm"` to every `actions/setup-node@v4` step in every job
3. **FR-003**: Remove `needs:` from `lint` and `typecheck`; add `needs: [lint, typecheck]` to `test`
4. **FR-005**: Delete `max-parallel: 1` from the Playwright strategy block
5. **FR-002**: Add Playwright browser cache (`actions/cache@v4`) before each `playwright install --with-deps` step

Push and verify all SC-001–SC-007 scenarios (see Validation section below).

---

### Tier 2 (FR-006–FR-008) — After Tier 1 passes CI

6. **FR-006**: Add Docker image caching sequence to all Supabase-starting jobs (`playwright`, `perf-gate`, `quickstart`). Follow the exact step ordering from `plan.md` (CLI install → version capture → cache restore → load → start → export on miss).
7. **FR-007**: Add the new `build` job with `needs: test` and `upload-artifact@v4`.
8. **FR-008**: Replace `pnpm build` in `playwright` and `perf-gate` with `download-artifact@v4` + integrity check. Update both jobs' `needs:` to reference `build`.

Push and verify SC-003, SC-004.

---

### Tier 3 (FR-009–FR-010) — After Tier 2 passes CI

9. **FR-009**: Remove `needs: integration` from `quickstart`; ensure `playwright` already uses `needs: build` (done in FR-008). No other YAML needed.
10. **FR-010**: Create a single `test` job that runs both `pnpm test:unit` and `pnpm test:integration` sequentially. Delete the existing `integration` job. Update `build`'s `needs:` to `test` (singular).

Push and verify SC-008.

## Gotchas

### Docker cache step ordering is mandatory
The Supabase CLI must be installed **before** its version string can be captured. The cache key depends on the version output. Never put the cache restore step before `setup-cli`.

```
✅ CORRECT: setup-cli → get version → restore cache → load → start → export on miss
❌ WRONG:   restore cache → setup-cli → get version → load → start → export on miss
```

### No `restore-keys` on Docker cache
A partial Supabase image archive causes `supabase start` to fail with cryptic "image not found" errors. Use exact key match only. Do not add a `restore-keys` fallback.

### `--with-deps` must always run for Playwright
The Playwright browser cache stores the browser binaries (~200 MB). OS-level system packages are **not** cached. `pnpm exec playwright install --with-deps` must run on every job, even on a cache hit — the `--with-deps` part installs OS packages via apt-get, which is fast on warm runners.

### Build-time env vars are safe as placeholders
The `build` job does not have a running Supabase instance. This is intentional. The project uses Next.js App Router with no static export — `NEXT_PUBLIC_SUPABASE_URL` is read at server startup, not baked into the JS bundle. Use `http://localhost:54321` as a build-time placeholder. Consumer jobs inject the real URL at server start via the existing `setsid bash -c "export NEXT_PUBLIC_SUPABASE_URL=..."` pattern.

### Artifact download path behavior
When `download-artifact@v4` downloads to `path: .next/`, files are placed directly under `.next/` (not under `.next/<artifact-name>/`). The resulting layout is:

```
workspace/
  .next/
    BUILD_ID          ← directly here
    server/
    static/
    ...
```

This is correct and matches what `pnpm start` expects. The integrity check (`[ -f .next/BUILD_ID ]`) will catch any path mismatch immediately.

### `quickstart` decoupling needs no extra mechanism
GitHub Actions automatically marks the workflow as FAILED when any job fails, regardless of that job's `needs:` relationships. Simply removing `needs: integration` from `quickstart` is sufficient. Do **not** add `continue-on-error: true` — that would hide failures and make the pipeline appear green when `quickstart` fails.

## Validation

### How to verify each success criterion

| SC | How to verify |
|---|---|
| **SC-001** (pnpm cache) | Push twice with no lockfile change. Second run log shows `Cache restored from key: setup-node-Linux-...`. No packages downloaded. |
| **SC-002** (Playwright browser cache) | Push twice with no `package.json` change. Second run shows `Cache hit` on the Playwright browser cache step. |
| **SC-003** (Docker cache) | Push twice with same Supabase CLI version. Second run shows `Cache hit` on `Restore Supabase Docker image cache` and `docker load` step runs. |
| **SC-004** (single build) | Inspect pipeline graph — exactly one job named `Build Next.js`. No `pnpm build` step in `playwright` or `perf-gate` jobs. |
| **SC-005** (parallel lint/typecheck) | Inspect the job list in GitHub Actions UI — `lint` and `typecheck` both show "In progress" simultaneously. |
| **SC-006** (parallel Playwright) | Inspect the job list — `Playwright UI (baseline)` and `Playwright UI (playtest)` both show "In progress" simultaneously. |
| **SC-007** (frozen lockfile) | On a test branch, manually add a dependency to `package.json` without running `pnpm install`. Push. The `test` job fails at `pnpm install --frozen-lockfile` with `ERR_PNPM_OUTDATED_LOCKFILE`. |
| **SC-008** (quickstart decoupled) | Trigger a run where `quickstart` fails (e.g., a Docker connectivity issue or a deliberate script error). Observe that `playwright` and `perf-gate` are NOT skipped — they start and run normally. But the overall workflow is marked `FAILED` (red). |

### Quick lint of YAML syntax (no push required)

```bash
# Install actionlint if not present
brew install actionlint   # macOS

# Run from repo root
actionlint .github/workflows/ci.yml
```

`actionlint` catches most YAML and GitHub Actions schema issues before pushing.
