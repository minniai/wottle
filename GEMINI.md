# Wottle - Project Context

## Project Overview

Wottle is a local-first board game prototype built with Next.js and Supabase. It focuses on a two-player experience with real-time matchmaking and gameplay. The project uses a spec-driven development workflow.

## Technology Stack

* **Frontend:** Next.js (App Router), React, Tailwind CSS.
* **Backend:** Supabase (PostgreSQL, Auth, Realtime).
* **Language:** TypeScript.
* **Testing:**
  * **Unit/Contract:** Vitest (`tests/unit`, `tests/contract`).
  * **Integration:** Vitest (`tests/integration`).
  * **E2E:** Playwright (`tests/integration/ui`).
  * **Performance:** Artillery (`tests/perf`).
* **State Management:** Zustand (implied by `package.json`).

## Key Commands

### Setup & Development

* **Full Setup:** `pnpm quickstart` (Runs Supabase preflight, starts Docker, seeds data, updates `.env.local`).
* **Start Dev Server:** `pnpm dev` (Starts Next.js on `localhost:3000`).

### Testing

* **Unit Tests:** `pnpm test:unit` (Runs unit and contract tests).
* **Integration Tests:** `pnpm test:integration` (Requires Supabase).
* **E2E Tests:** `pnpm exec playwright test` (Runs browser automation; requires Supabase + Next.js).
* **Performance Tests:** `pnpm perf:swap`, `pnpm perf:lobby-presence`, etc.

### Database

* **Migrate:** `pnpm supabase:migrate`
* **Seed:** `pnpm supabase:seed`
* **Reset:** `pnpm supabase:reset`

## Project Structure

* **`app/`**: Next.js App Router pages and API routes.
* **`components/`**: React components (UI, Game Board, Lobby).
* **`lib/`**: Core logic, utilities, backend services, and constants.
* **`specs/`**: GitHub Spec-kit specifications (e.g., `002-two-player-playtest`).
* **`tests/`**: Test suites organized by type (unit, integration, contract, perf).
* **`scripts/`**: Utility scripts for database management, seeding, and verification.
* **`supabase/`**: Supabase configuration, migrations, and policies.

## Development Conventions

* **Spec-Driven:** Development follows detailed specifications located in `specs/`.
* **TDD:** Test-Driven Development is strictly followed. Write tests (Red) before implementation (Green).
* **Feature Flags:** UI features often controlled by flags (e.g., `NEXT_PUBLIC_ENABLE_PLAYTEST_LOBBY`).
* **Environment:** relies on `.env.local` for secrets (generated by `quickstart`).
